# Titan007 比赛列表与实时更新数据结构

本文记录 Titan007 足球即时比分旧版列表页使用的初始比赛数据和实时更新协议。
字段结论来自 2026-07-19 实际页面及页面加载的 `func.js`、`commonWs.js` 和
`SocketHelper.js`。这些是未公开的网页内部接口，源站可能在不通知的情况下调整
字段、频道或传输格式；采集端必须保留格式和数量校验。

## 1. 数据入口

列表页：

```text
https://live.titan007.com/oldIndexall.aspx
```

初始比赛快照由页面启动链加载：

```text
https://livestatic.titan007.com/vbsxml/bfdata_ut.js?r={动态值}
```

实时变化通过以下 WebSocket 提供：

```text
wss://live.titan007.com/stream
```

旧版完整列表页订阅 `change_ut_xml`。首页或机构赔率页还可能订阅：

- `ch_goalBf3_xml2`：首页使用的公司 3 赔率变化；
- `ch_goal{company_id}_xml2`：指定机构页面的赔率变化。

## 2. `bfdata_ut.js` 初始快照

每场比赛是一个以 `^` 分隔的数组：

```javascript
A[n] = "字段0^字段1^字段2^...".split("^");
```

数组下标从 `0` 开始。当前比赛 ID 脚本只提取 `A[n][0]`，并使用
`matchcount` 校验唯一正整数 ID 的数量；它尚未保存本节列出的其他字段。

### 2.1 比赛身份、队名和比分

| 下标 | 含义 | 建议字段 |
|---:|---|---|
| `A[n][0]` | Titan007 比赛 ID | `match_id` |
| `A[n][2]` | 联赛简体名称 | `league_name` |
| `A[n][3]` | 联赛繁体名称 |  |
| `A[n][5]` | 主队简体名称 | `home_team` |
| `A[n][6]` | 主队繁体名称 |  |
| `A[n][8]` | 客队简体名称 | `away_team` |
| `A[n][9]` | 客队繁体名称 |  |
| `A[n][14]` | 主队当前全场比分 | `home_score` |
| `A[n][15]` | 客队当前全场比分 | `away_score` |
| `A[n][16]` | 主队半场比分 | `home_half_score` |
| `A[n][17]` | 客队半场比分 | `away_half_score` |

简体列表页调用 `getTeamName(i, 1, 3, 1)` 和
`getTeamName(i, 2, 3, 1)`，最终分别使用 `A[n][5]` 和 `A[n][8]`。
队名中可能带有源站 HTML，例如标识中立场的 `<font>...</font>`；落库前应做
受控的 HTML 清理，同时保留 `(中)` 语义。

### 2.2 时间和状态

| 下标 | 含义 | 备注 |
|---:|---|---|
| `A[n][11]` | 计划开赛时刻 | 页面使用的 `HH:mm` 部分 |
| `A[n][12]` | 当前比赛阶段的计时基准 | 六段式日期时间，页面用它计算实时分钟 |
| `A[n][13]` | 比赛状态码 | 状态映射见下表 |
| `A[n][36]` | 比赛日期 | 页面排序和组合计划开赛时间时使用 |
| `A[n][43]` | 时间换算相关字段 | 传给页面的 `AmountTimeDiff` |
| `A[n][56]` | 补时分钟 | 非空时在时间列显示“补 x'” |

状态码映射：

| 状态码 | 含义 |
|---:|---|
| `-14` | 推迟 |
| `-13` | 中断 |
| `-12` | 腰斩 |
| `-11` | 待定 |
| `-10` | 取消 |
| `-1` | 完场 |
| `0` | 未开始；网页状态单元格显示为空 |
| `1` | 上半场 |
| `2` | 中场 |
| `3` | 下半场 |
| `4` | 加时 |
| `5` | 点球 |

网页展示的 `50`、`90+9` 等比赛分钟不是独立源字段：

- 上半场根据 `A[n][12]` 与网页服务器时间差计算；超过 45 分钟显示
  `45+x`；
- 下半场根据 `A[n][12]` 计算后加 46；超过 90 分钟显示 `90+x`；
- 页面把补时显示上限限制为 15 分钟；
- `A[n][56]` 是源站给出的补时值，显示在计划时间列，不等于实时分钟文本。

因此建议分别保存原始状态码、规范化状态文本和可选的计算分钟，不要把
`90+9` 当成状态码：

```text
state_code = 3
status_text = 下半场
match_minute_text = 90+9
```

### 2.3 其他会被实时更新的快照字段

| 下标 | 含义 |
|---:|---|
| `A[n][18]` / `A[n][19]` | 主队 / 客队红牌数 |
| `A[n][20]` / `A[n][21]` | 主队 / 客队黄牌数 |
| `A[n][27]` | 是否已有阵容信息 |
| `A[n][30]` | 附加说明文本 |
| `A[n][42]` | 结构化比赛说明 |
| `A[n][48]` / `A[n][49]` | 主队 / 客队角球数 |
| `A[n][50]` | 角球数据显示标记 |
| `A[n][54]` / `A[n][55]` | VAR/视频说明简体 / 繁体文本 |
| `A[n][68]` | VAR 说明关联的球队 ID |

## 3. WebSocket 传输格式

页面向 `wss://live.titan007.com/stream` 连接并订阅频道。服务端消息是压缩的
二进制数据；网页按以下顺序解码：

1. `Blob` 转为 `ArrayBuffer`；
2. 使用 `pako.inflate(..., {to: "string"})` 解压；
3. 将解压文本解析为 JSON；
4. 读取 JSON 中与订阅频道同名的属性。

概念上的消息信封如下：

```json
{
  "change_ut_xml": "一场比赛更新!另一场比赛更新",
  "ch_goalBf3_xml2": "可选的赔率更新"
}
```

实际消息不保证同时包含两个属性。旧版完整列表页通常只订阅和处理
`change_ut_xml`。

## 4. `change_ut_xml` 比赛实时更新

`change_ut_xml` 的值是字符串：多场比赛用 `!` 分隔，每场的字段用 `^`
分隔。

```text
match0_field0^match0_field1^...!match1_field0^match1_field1^...
```

网页将一场比赛解析为 `D = item.split("^")`。当前处理器读取以下字段：

| 下标 | 含义 | 回写的初始快照字段 |
|---:|---|---|
| `D[0]` | 比赛 ID | 用来定位对应的 `A[n]` |
| `D[1]` | 状态码 | `A[n][13]` |
| `D[2]` | 主队全场比分 | `A[n][14]` |
| `D[3]` | 客队全场比分 | `A[n][15]` |
| `D[4]` | 主队半场比分 | `A[n][16]` |
| `D[5]` | 客队半场比分 | `A[n][17]` |
| `D[6]` | 主队红牌数 | `A[n][18]` |
| `D[7]` | 客队红牌数 | `A[n][19]` |
| `D[8]` | 计划开赛时刻 | `A[n][11]` |
| `D[9]` | 当前阶段计时基准 | `A[n][12]` |
| `D[10]` | 附加说明文本 | `A[n][30]` |
| `D[11]` | 阵容信息标记 | `A[n][27]` |
| `D[12]` | 主队黄牌数 | `A[n][20]` |
| `D[13]` | 客队黄牌数 | `A[n][21]` |
| `D[14]` | 比赛日期 | 与 `D[8]` 组合用于页面排序 |
| `D[15]` | 结构化比赛说明 | `A[n][42]` |
| `D[16]` | 主队角球数 | `A[n][48]` |
| `D[17]` | 客队角球数 | `A[n][49]` |
| `D[18]` | 角球数据显示标记 | `A[n][50]` |
| `D[19]` | VAR/视频说明简体文本 | `A[n][54]` |
| `D[20]` | VAR/视频说明繁体文本 | `A[n][55]` |
| `D[21]` | 补时分钟 | `A[n][56]` |
| `D[22]`～`D[24]` | 当前页面处理器未读取 | 视为保留字段 |
| `D[25]` | VAR 说明关联的球队 ID | `A[n][68]` |

`D[15]` 不是普通状态字符串。页面将它继续按 `|` 和 `;` 拆分，用于展示
电视/购买入口、先开球队、角球、90/120 分钟比分、两回合比分、加时和点球
等复合说明。因此采集时应先保存原文，除非已有覆盖这些变体的解析测试。

WebSocket 更新不包含主客队名称和联赛名称；这些静态身份字段来自
`bfdata_ut.js`，实时消息用 `D[0]` 关联比赛。

## 5. 可选赔率频道

首页或机构页面会额外处理 `ch_goalBf3_xml2` 或
`ch_goal{company_id}_xml2`。其值采用：

```text
比赛赔率记录!比赛赔率记录$$维护标记
```

每条比赛赔率记录内部用逗号分隔，包含：

- 比赛 ID；
- 常规时间的亚让盘口与主客赔率；
- 常规时间的进球数盘口与大小球赔率；
- 常规时间的胜平负赔率；
- 走地状态和各市场封盘标记；
- 加时阶段对应的亚让、进球数、胜平负赔率；
- 点球阶段对应的亚让、进球数、胜平负赔率。

该频道用于列表页展示当前报价，不是赔率变动历史接口，不能替代项目现有的
赔率变化页面采集。

## 6. 断线降级

WebSocket 达到最大重连次数或异常关闭后，网页退回 HTTP 轮询：

- 每约 2 秒检查 `vbsxml/time.txt`；
- 检测到版本变化后请求 `vbsxml/change_ut.xml`；
- 周期性请求 `vbsxml/change2_ut.xml` 做补偿同步。

因此“网页上的实时比分”是初始 `bfdata_ut.js` 快照加后续 WebSocket（或降级
轮询）更新的结果。只抓一次 `bfdata_ut.js` 不能保证长时间运行时的比分和状态
持续准确。
